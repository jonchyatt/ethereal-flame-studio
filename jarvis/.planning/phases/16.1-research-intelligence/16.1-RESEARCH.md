# Phase 16.1: Research Intelligence - Research

**Researched:** 2026-03-17
**Domain:** Outcome tracking, expiry monitoring, cross-domain inference, confidence evolution -- all via SQL + cron + event hooks
**Confidence:** HIGH

## Summary

Phase 16.1 adds intelligence to the existing research library without ML, embeddings, or new dependencies. The research_entries table (Phase 12) and research MCP tools (Phase 16) already store and retrieve structured findings. The approval_audit table (Phase 15) already logs outcomes of grant applications and bill payments. The scheduled_tasks system (Phase 12) with its handler registry and 60-second hot-reload already runs periodic jobs. Phase 16.1 connects these pieces with: (1) a junction table linking approval outcomes to the research entries that informed them, (2) SQL queries that flag expired entries and prioritize re-research, (3) event hooks in the grant workflow that create cross-domain entries, and (4) simple rules that upgrade/downgrade confidence based on corroboration and contradiction.

The self-improvement pipeline (Phase D) established the pattern: a scheduled cron handler runs analysis, produces actionable changes, and logs results. Research intelligence follows the same model. A `research-intelligence` cron handler runs daily, scans for stale entries, evolves confidence scores, and generates cross-domain inferences. Event-driven hooks in the grant workflow produce immediate feedback (outcome linking, cross-domain entry creation) while the cron handles batch operations (expiry scanning, confidence evolution).

**Primary recommendation:** Add one junction table (`research_outcomes`), one new cron handler (`research-intelligence`), three new functions in researchStore.ts (stale entry detection, confidence evolution, cross-domain inference), and two event hooks in grantApplicationWorkflow.ts (outcome linking after approval result, cross-domain entry creation after successful submission). Two new MCP tools (`track_outcome` and `get_research_insights`) expose intelligence to the user.

## Architecture Patterns

### Pattern 1: Junction Table for Outcome Tracking (INTEL-01)

**What:** A `research_outcomes` table links `approval_audit.id` to one or more `research_entries.id` records, recording which research informed which application outcome.

**Why not a metadata field:** The approval_audit.metadata JSON already stores `grantName` and `entityName`, but cannot be efficiently queried for "which research entries led to approved applications?" A proper junction table enables SQL JOINs.

**Schema:**
```typescript
export const researchOutcomes = sqliteTable('research_outcomes', {
  id: integer('id').primaryKey({ autoIncrement: true }),
  approvalId: text('approval_id').notNull(),     // FK to approval_audit.id
  researchEntryId: integer('research_entry_id').notNull(), // FK to research_entries.id
  outcome: text('outcome').notNull(),             // 'approved' | 'rejected' | 'pending' | 'submitted'
  domain: text('domain').notNull(),               // 'grant' | 'credit' (for quick filtering)
  topic: text('topic').notNull(),                 // Denormalized for fast queries
  notes: text('notes'),
  createdAt: text('created_at').notNull().$defaultFn(() => new Date().toISOString()),
});
```

**Why denormalize domain+topic:** Avoids a JOIN back to research_entries for the most common query pattern ("show me all approved grant outcomes"). The junction table is append-only (outcomes don't change), so denormalization is safe.

**Event hook location:** In `grantApplicationWorkflow.ts`, after Stage 4 (approval result known), link all grant research entries to the approval_audit record:
```typescript
// After approval result (Stage 4)
const grantEntries = await getResearchByTopic('grant', grantName);
const entityEntries = await getResearchByTopic('business', entityName);
for (const entry of [...grantEntries, ...entityEntries]) {
  await linkOutcome(approvalResult.approvalId, entry.id, approvalResult.status, 'grant', grantName);
}
```

### Pattern 2: Expiry Monitoring via Cron (INTEL-02)

**What:** A scheduled task runs daily, queries `research_entries WHERE expires_at < NOW() AND confidence != 'expired'`, marks them as stale, and generates a Telegram notification for high-value expired entries.

**High-value definition:** An entry is high-value if:
1. It has linked outcomes where outcome = 'approved' (research that led to success), OR
2. Its tags contain "urgent" or "high-priority", OR
3. Its domain is 'grant' and fieldName is 'deadline' (time-sensitive by nature)

**Stale handling:**
- Set `confidence = 'expired'` (new confidence level, backwards-compatible since confidence is free-form text)
- Add note: `"Auto-expired on {date}. Previous confidence: {old_confidence}"`
- Do NOT delete -- expired data is still queryable and useful for historical analysis

**Re-research notification:** Send a Telegram message listing high-value expired entries with the suggestion "Re-research these topics: [list]". The user (or a future auto-research loop) initiates the actual re-research.

**Cron schedule:** Daily at 6:00 AM ET (after the 5:00 AM reflection loop, before Jon's morning briefing).

### Pattern 3: Cross-Domain Inference Rules (INTEL-03)

**What:** Explicit if/then rules that create new research entries when certain conditions are met. Not ML -- just deterministic mappings.

**Rules array:**
```typescript
const CROSS_DOMAIN_RULES: InferenceRule[] = [
  {
    trigger: { domain: 'grant', outcome: 'approved' },
    creates: {
      domain: 'credit',
      fieldName: 'revenue_evidence',
      fieldValueTemplate: 'Approved grant: {topic} ({amount}). Demonstrates revenue pipeline for credit applications.',
      confidence: 'medium',
      tags: '["auto-inferred","grant-to-credit"]',
    },
  },
  {
    trigger: { domain: 'grant', outcome: 'approved' },
    creates: {
      domain: 'business',
      fieldName: 'grant_track_record',
      fieldValueTemplate: 'Successfully awarded: {topic}. Strengthens future grant applications.',
      confidence: 'high',
      tags: '["auto-inferred","grant-history"]',
    },
  },
  {
    trigger: { domain: 'credit', fieldName: 'approval_status', fieldValue: 'approved' },
    creates: {
      domain: 'grant',
      fieldName: 'financial_stability_evidence',
      fieldValueTemplate: 'Approved for credit: {topic}. Demonstrates financial stability for grant applications.',
      confidence: 'medium',
      tags: '["auto-inferred","credit-to-grant"]',
    },
  },
];
```

**When rules fire:**
1. **Event-driven (immediate):** After grantApplicationWorkflow Stage 6 (successful submission confirmed), run inference rules synchronously.
2. **Batch (cron):** The daily intelligence handler scans recent outcomes (last 24h) and runs inference rules for any that were missed.

**Deduplication:** Before creating an inferred entry, check if one with the same domain+topic+fieldName+tags containing "auto-inferred" already exists. Skip if duplicate.

### Pattern 4: Confidence Evolution (INTEL-04)

**What:** Simple deterministic rules that adjust confidence based on evidence patterns.

**Rules:**
1. **Corroboration (upgrade):** If two or more entries exist for the same domain+topic+fieldName with the same fieldValue but different sources, upgrade both to `confidence='high'`.
2. **Contradiction (downgrade):** If two entries for the same domain+topic+fieldName have different fieldValues, downgrade both to `confidence='low'` and add note: `"Conflicts with entry #{otherEntryId}: {otherValue}"`.
3. **Age decay:** Entries older than 90 days with no linked outcomes and no updates get downgraded one level (high->medium, medium->low). Not applied to entries with `confidence='expired'` (already handled by expiry).
4. **Outcome boost:** Entries linked to approved outcomes get upgraded to `confidence='high'` (success validates the research).
5. **Outcome penalty:** Entries linked to rejected outcomes where the rejection reason mentions the field get note: `"Linked to rejected application -- verify accuracy"` (no automatic downgrade, just flagging).

**Execution:** All confidence evolution runs in the daily cron handler. Group entries by domain+topic+fieldName, apply rules in order (corroboration, contradiction, age decay, outcome boost).

### Pattern 5: Intelligence Loop as Cron Handler

**What:** Register a `research-intelligence` handler in cronRunner.ts HANDLERS, following the exact pattern of `daily-reflection`.

**Handler flow:**
```
1. Scan for expired entries -> mark stale, collect high-value list
2. Run confidence evolution across all entries
3. Run cross-domain inference for recent outcomes (last 24h)
4. Send Telegram summary: "Research Intelligence: {N} entries expired, {M} confidence changes, {K} cross-domain inferences created. High-value re-research needed: [list]"
5. Return summary string for lastResult logging
```

**Seed as system task:** Add to `seedSystemTasks()` alongside daily-reflection.

### Anti-Patterns to Avoid

- **ML/embeddings for intelligence:** The research library has hundreds of entries, not millions. SQL queries with clear rules produce better results than statistical models at this scale. Embeddings are for retrieval (already covered by keyword search), not intelligence.
- **Auto-re-research without human trigger:** Automatically kicking off sub-agent research runs from a cron is expensive (API costs) and potentially unwanted. The cron should NOTIFY about stale entries, not automatically re-research them.
- **Modifying approval_audit:** The audit table is a legal record. Never modify it. The junction table (research_outcomes) is the right place for outcome-to-research links.
- **Complex confidence scoring (0-100):** The existing confidence levels are 'high'|'medium'|'low'. Adding numeric scores creates complexity without value. Keep the levels, add 'expired' as a fourth state.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Outcome tracking | Custom event bus | Junction table + direct function calls in workflow | Event buses add complexity for 2 event types. Direct calls are debuggable. |
| Stale detection | In-memory timer per entry | SQL query in cron | Entries persist across restarts. SQL is the source of truth. |
| Intelligence scheduling | setInterval in researchStore | cronRunner handler registry | Handler registry already exists, has hot-reload, logging, error handling. |
| Cross-domain reasoning | LLM-based inference | Deterministic rule array | Rules are predictable, auditable, and free. LLM inference is expensive and unpredictable. |

## Common Pitfalls

### Pitfall 1: Confidence Column is Free-Form Text
**What goes wrong:** The `confidence` column is `text('confidence')` with a default of 'medium'. Adding 'expired' as a new value works, but any code that does `if (confidence === 'high' || confidence === 'medium' || confidence === 'low')` will miss expired entries.
**How to avoid:** Grep for all confidence comparisons in the codebase and ensure they handle 'expired'. The form-filler's eligibility check (Stage 2) already checks `confidence === 'low'` -- 'expired' entries should be treated similarly.

### Pitfall 2: Circular Inference
**What goes wrong:** A grant approval creates a credit entry (rule 1). If a credit approval then creates a grant entry (rule 3), and that grant entry gets linked to a new grant application, the inference chain could loop.
**How to avoid:** Inferred entries are tagged with `"auto-inferred"`. Inference rules NEVER fire on entries tagged `"auto-inferred"`. This breaks the cycle.

### Pitfall 3: Duplicate Outcome Links
**What goes wrong:** If a grant workflow is retried (user re-runs after fixing entity fields), the same research entries get linked to a new approval_audit record. This is actually correct behavior (different application attempt), but could inflate "successful research" counts.
**How to avoid:** Each research_outcomes row links to a specific approval_audit.id. Multiple rows for the same research_entry_id are fine -- they represent different application attempts. Queries should count distinct approval_audit.id, not rows.

### Pitfall 4: Cron Race with Workflow
**What goes wrong:** The daily cron marks an entry as 'expired' while a grant workflow is actively using it for eligibility checks.
**How to avoid:** The cron runs at 6 AM. Grant workflows are triggered by user interaction. In practice these won't overlap. If they do, the workflow queries the entry at a point in time -- it gets the confidence at query time. The cron updating it afterward is fine; the next workflow run will see the updated confidence.

### Pitfall 5: Overwriting User-Set Confidence
**What goes wrong:** Jon manually sets an entry to 'high' confidence via store_research. The cron's age decay rule downgrades it to 'medium' 90 days later.
**How to avoid:** Add a `manualOverride` boolean field to research_entries (or check if source contains 'manual' or 'user'). Entries with manual confidence should not be auto-decayed. Simpler: only age-decay entries where source is a URL (automated research). Entries sourced from entity profiles or manual input are excluded from age decay.

### Confidence Evolution (Corroboration Check)
```typescript
// Group entries by domain+topic+fieldName, check for matching values from different sources
export async function evolveConfidence(): Promise<{ upgraded: number; downgraded: number; decayed: number }> {
  const db = getDataDb();
  const allEntries = await db.select().from(researchEntries)
    .where(not(eq(researchEntries.confidence, 'expired')));

  const groups = new Map<string, ResearchEntry[]>();
  for (const entry of allEntries) {
    const key = `${entry.domain}::${entry.topic}::${entry.fieldName}`;
    const group = groups.get(key) || [];
    group.push(entry);
    groups.set(key, group);
  }

  let upgraded = 0, downgraded = 0, decayed = 0;

  for (const [, entries] of groups) {
    if (entries.length < 2) continue;

    // Check corroboration: same value, different sources
    const valueGroups = new Map<string, ResearchEntry[]>();
    for (const e of entries) {
      const v = e.fieldValue?.toLowerCase().trim() || '';
      const vg = valueGroups.get(v) || [];
      vg.push(e);
      valueGroups.set(v, vg);
    }

    if (valueGroups.size === 1) {
      // All agree -- check if different sources
      const sources = new Set(entries.map(e => e.source).filter(Boolean));
      if (sources.size >= 2) {
        // Corroborated -- upgrade all
        for (const e of entries) {
          if (e.confidence !== 'high') {
            await updateResearchEntry(e.id, { confidence: 'high' });
            upgraded++;
          }
        }
      }
    } else if (valueGroups.size > 1) {
      // Contradiction -- downgrade all
      for (const e of entries) {
        if (e.confidence !== 'low') {
          await updateResearchEntry(e.id, {
            confidence: 'low',
            notes: `Contradicted by other sources for same field`
          });
          downgraded++;
        }
      }
    }
  }

  return { upgraded, downgraded, decayed };
}
```

### Cross-Domain Inference
```typescript
// In researchIntelligence.ts
export async function runCrossDomainInference(): Promise<number> {
  const db = getDataDb();
  const oneDayAgo = new Date(Date.now() - 24 * 60 * 60 * 1000).toISOString();

  // Get recent approved outcomes
  const recentOutcomes = await db.select()
    .from(researchOutcomes)
    .where(and(
      eq(researchOutcomes.outcome, 'approved'),
      gt(researchOutcomes.createdAt, oneDayAgo)
    ));

  let created = 0;
  for (const outcome of recentOutcomes) {
    for (const rule of CROSS_DOMAIN_RULES) {
      if (outcome.domain !== rule.trigger.domain) continue;

      // Check deduplication
      const existing = await searchResearch(
        outcome.topic,
        rule.creates.domain
      );
      const isDuplicate = existing.some(e =>
        e.fieldName === rule.creates.fieldName &&
        e.tags?.includes('auto-inferred')
      );
      if (isDuplicate) continue;

      // Create inferred entry
      await saveResearchEntry({
        domain: rule.creates.domain,
        topic: outcome.topic,
        fieldName: rule.creates.fieldName,
        fieldValue: rule.creates.fieldValueTemplate
          .replace('{topic}', outcome.topic),
        source: `auto-inferred from ${outcome.domain} outcome`,
        confidence: rule.creates.confidence,
        tags: rule.creates.tags,
      });
      created++;
    }
  }
  return created;
}
```

## Open Questions

1. **Should age decay apply to entity profile entries?**
   - What we know: Entity profiles are loaded from markdown with source='grant-entity-profiles.md'. They represent relatively stable data (EIN, entity type, mission).
   - What's unclear: Should a 90-day-old EIN entry be decayed? Probably not.
   - Recommendation: Exclude entries with source='grant-entity-profiles.md' from age decay. Entity profiles are refreshed via load_entity_profiles, not age decay.

2. **Should the `track_outcome` tool support non-grant domains?**
   - What we know: Currently only grantApplicationWorkflow produces approval_audit records with grant metadata.
   - What's unclear: Will credit applications go through the same approval gateway?
   - Recommendation: Make track_outcome generic (accept any domain). When credit workflows exist, they will use the same pattern.

3. **Notification frequency for stale entries**
   - What we know: Daily cron runs at 6 AM. Could produce daily "stale entries" notifications.
   - What's unclear: Will this become noisy if many entries expire?
   - Recommendation: Only notify once per entry (add a `staleNotifiedAt` field or check notes for "Auto-expired" marker). Re-notify only if the entry is high-value and has been stale for 7+ days without re-research.
