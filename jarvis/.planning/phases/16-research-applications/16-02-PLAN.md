**Create `src/lib/jarvis/data/entityProfiles.ts`:**

This module parses the entity profiles markdown file and loads each field as a separate `research_entries` row with `domain='business'`.

```typescript
/**
 * Entity Profile Loader -- Jarvis v5.0
 *
 * Parses grant-entity-profiles.md and loads each field as a research_entries row.
 * Each venture becomes a topic (e.g., 'Satori Living Foundation').
 * Each field becomes a separate row (fieldName='ein', fieldValue='93-3978916').
 *
 * [FILL] placeholders are stored with confidence='low' and notes='MISSING - needs human input'.
 * This ensures the form-filler can query entity data and knows which fields are incomplete.
 *
 * Re-running the loader clears and re-imports (idempotent via deleteResearchTopic first).
 */

import * as fs from 'fs/promises';
import * as path from 'path';
import { saveResearchEntry, deleteResearchTopic } from '../research/researchStore';
```

Export constants:
- `ENTITY_PROFILES_PATH`: `path.resolve(__dirname, '../../../../.paul/concepts/grant-entity-profiles.md')` -- this resolves from `src/lib/jarvis/data/` up to project root then into `.paul/concepts/`. Verify this path works on Windows.
- Alternative: Use `path.join(process.cwd(), '.paul/concepts/grant-entity-profiles.md')` which is simpler and works when cwd is project root.

Use `process.cwd()` approach: `export const ENTITY_PROFILES_PATH = path.join(process.cwd(), '.paul/concepts/grant-entity-profiles.md');`

Export `loadEntityProfiles()` async function that:

1. Reads the markdown file from `ENTITY_PROFILES_PATH`
2. Parses venture sections by looking for `## VENTURE` headings (the file uses `## VENTURE A: Satori Living Foundation (Longevity Nonprofit)` format)
3. For each venture, extracts the entity name from the heading (e.g., "Satori Living Foundation")
4. Within each venture, parses `- Field name: value` lines
5. Before inserting, calls `deleteResearchTopic('business', entityName)` to clear stale data (idempotent)
6. For each field, calls `saveResearchEntry()` with:
   - `domain`: `'business'`
   - `topic`: entity name (e.g., `'Satori Living Foundation'`)
   - `fieldName`: normalized field name (lowercase, spaces to underscores, e.g., `'legal_business_name'`, `'ein'`, `'annual_revenue_2024'`)
   - `fieldValue`: the value after the colon, trimmed
   - `source`: `'grant-entity-profiles.md'`
   - `confidence`: `'low'` if value contains `[FILL]` or is empty, `'high'` otherwise
   - `notes`: `'MISSING - needs human input'` if `[FILL]`, else null
   - `tags`: `'["entity-profile"]'`
7. Returns a summary: `{ venturesLoaded: number, fieldsStored: number, missingFields: number }`

Parsing rules:
- Venture heading pattern: `/^## VENTURE [A-Z]: (.+?)(?:\s*\(.+\))?\s*$/`
- Extract entity name from capture group 1 (e.g., "Satori Living Foundation")
- Section heading pattern: `/^### (.+)$/` (e.g., "LEGAL/REGISTRATION", "FINANCIAL PROFILE")
- Field pattern: `/^- (.+?):\s*(.*)$/`
- Normalize field name: `sectionName + '_' + fieldLabel` lowercased with spaces/punctuation replaced by underscores. Example: section "LEGAL/REGISTRATION", field "EIN" -> `"legal_registration_ein"`. Keep it simple -- just the field label is fine too: `"ein"`, `"entity_type"`, `"annual_revenue_2023"`.
- Actually, simpler approach: just use the field label directly, normalized. Section context is captured by the field name itself (e.g., "Mission statement (1 sentence)" -> `"mission_statement_1_sentence"`). Skip the section prefix to avoid overly long field names.
- Multi-line values: if a line after a field starts with `>` or doesn't match the field pattern and isn't a heading, append it to the previous field's value.
- Skip the "HOW TO USE THIS FILE" section and "Grant History Tracker" section (not entity data).
