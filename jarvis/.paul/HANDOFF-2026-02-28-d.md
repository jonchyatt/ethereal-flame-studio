## Key Decisions Made

- **Embedding cleanup on delete_all is mandatory** — orphaned vectors corrupt future searches
- **Pre-synthesized content eliminates double API cost** — consolidation confirm phase now accepts cached preview results
- **`findMemoriesMatching` is the canonical search entry point** — all callers should use it, not reach into vectorSearch directly
- **Anthropic singleton follows established pattern** — OpenAI client already uses lazy singleton in embeddings.ts; applied same to consolidation + summarization
- **Systemic issues documented for v4.1 scope:**
  1. In-memory BM25 should use SQLite FTS5 (loads all entries per search)
  2. Sequential embedding API should use OpenAI batch endpoint (50 calls → 1)
  3. Circular dep between memoryEntries↔vectorSearch solved by `require()` hack
  4. No embedding model versioning (model change = incompatible stored embeddings)
  5. Fire-and-forget embedding ops lack error aggregation
  6. SQL string interpolation in raw queries (not exploitable but bad practice)
